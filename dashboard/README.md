# Paris Apartment Dashboard — How to use & update

Open [dashboard.html](dashboard.html) in any browser (double-click in Finder works — no server needed).

## What's in this folder

| File | Role |
|---|---|
| `dashboard.html` | The UI. Self-contained — only depends on `listings.js`. |
| `listings.js` | Data file the browser loads. Generated by `build_dashboard.py`. |
| `listings.json` | Same data as JSON. For external tooling or inspection. |

## What it shows

For each listing:
- **Verdict** — the price per m² compared to recent DVF transactions in the same arrondissement + surface band:
  - `underpriced` — below 92% of the IQR floor (p25)
  - `below-band` — under the p25 of recent sales (usually a real handicap: north exposure, condition, low floor)
  - `in-band` — inside the IQR (p25–p75) — fair price for the size class and arrondissement
  - `above-band` — above p75 — typical for renovated or exceptional units
  - `overpriced` — more than 8% above p75 — needs strong condition/floor/view justification
  - `needs-data` — listing is missing critical info (surface or price)
- **Δ vs median** — % difference between the asking €/m² and the DVF median for the same surface band
- **Click any row** to expand for the full expert analysis, listing details, and link to the original ad

## Filtering

Hold `Cmd` (Mac) / `Ctrl` (Windows) to multi-select arrondissements or verdicts. Click any column header to sort.

## How to add new listings

1. **Forward the listing email to Claude**, or paste the listing URL into your next conversation.
2. Claude will:
   - Pull DVF for the new arrondissement if not already present (the per-commune CSV at `https://files.data.gouv.fr/geo-dvf/latest/csv/{year}/communes/75/751{arr_code}.csv` — note the `751` prefix, not `750`).
   - Append a new entry to `listings_input.json` (in the parent folder).
   - Rerun `build_dashboard.py` to regenerate `listings.js` and `listings.json`.
3. **Refresh `dashboard.html`** in the browser. Done.

## Agent-augmented enrichment (used on intake)

When new listings come in, Claude spawns up to **3 parallel sub-agents** to do work that would otherwise dump huge tool outputs into the main conversation:

```
intake (parallel)                analysis (parent)       output
─────────────────────────        ─────────────────       ──────
fetch URLs       ┐
extract photos   ├──► aggregated stubs ──► build_dashboard.py ──► dashboard
extra research   ┘
```

| Sub-agent | Purpose | Why parallel/sub |
|---|---|---|
| URL fetcher (Explore) | Pull full listing pages, extract structured fields | A single listing page can be 10-50k tokens of HTML — the agent returns just the 200 tokens we need |
| Image condition assessor | Download lead photos, view them, return condition grade + signals | Image binaries stay out of parent context; per-listing assessment scales without inflating main conversation |
| Chart builder | Generate matplotlib distribution PNGs | Self-contained Python work that doesn't need parent state |

What stays in the parent (no sub-agent):
- DVF cleaning pipeline (already lean Python)
- The build_dashboard.py orchestration (needs LISTINGS state)
- listings_input.json edits (need to integrate sub-agent findings)

**When NOT to use sub-agents:**
- Trivial work (overhead > savings)
- Tasks that need parent's in-flight state
- Anything that produces a small predictable output already

## Editing a listing yourself

If you want to add info (visit notes, AG minutes findings, updated condition, negotiation results), the cleanest path is to edit the listing dict in `build_dashboard.py` and rerun. The relevant fields:

| Field | Example | Notes |
|---|---|---|
| `condition` | `"perfect condition"` / `"renovated 2018"` / `"needs full gut"` | Drives the strengths/concerns section |
| `dpe_class` | `"D"` | Triggers an automatic concern note for D-G |
| `charges_yr_eur` | `4582` | Surfaces in the meta panel |
| `taxe_fonciere_eur` | `3335` | Surfaces in the meta panel |
| `features` | `["double exposure", "fireplace"]` | First 3 surface in the analysis |
| `notes` | free-form string | Pasted verbatim at the end of the analysis |

## Methodology in one paragraph

DVF (Demandes de Valeurs Foncières) is the French open-data registry of all recorded property transactions. The pipeline pulls per-arrondissement CSVs, drops mixed-lot sales (apartment + parking/cellar/shop) because those inflate €/m², groups by mutation ID to handle multi-row sales, trims outliers (< €5k/m² or > €40k/m²), and computes per-surface-band medians and IQRs. For each listing, the asking €/m² is positioned against the median + IQR of the most relevant surface band (≥80 m² listings use the 80–250 m² combined band for sample-size robustness; <80 m² listings use the <80 m² band).

What the pipeline does NOT do: distinguish unit condition, floor, exposure, or view. Two units in the same building with very different finishes look identical in DVF. So the **verdict** is best read as "is the seller asking inside the recorded-transaction range for this size + arrondissement?" — not "is the unit worth this." Condition is the dominant variable, and only an on-site visit settles it.
