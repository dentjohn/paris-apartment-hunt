1. Duplicate const ppm in renderRow() — SyntaxError blocked all dashboard JS

Priority: 10/10 Confidence: 10/10 JavaScript FIXED
J
Jason
JavaScript Specialist

Why this is a bug:

Two const ppm declarations exist in the same scope inside renderRow(l, idx) (dashboard_template.html lines 515 and 523 of the pre-fix version). const redeclaration is a SyntaxError at parse time, which causes the entire IIFE to refuse to execute. The static HTML (header, controls, instructions block) still rendered but the table body stayed empty because the script never ran. The bug was introduced in the "Highlight genuine value-candidate rows" commit when a second const ppm = l.price_per_m2 || 0 was added without renaming the existing display-only declaration.

Suggested fix:

Rename the two variables to disambiguate their purpose: ppmRaw for the numeric value used in comparisons, ppmDisplay for the string used in the cell.

Fix Prompt:

In dashboard_template.html, inside the renderRow() function, replace the two `const ppm = ...` declarations with one `const ppmRaw = l.price_per_m2 || 0` and one `const ppmDisplay = l.price_per_m2 ? \`€${l.price_per_m2.toLocaleString()}\` : '—'`. Update all downstream references: the value-candidate test uses `ppmRaw`; the rendered cell uses `${ppmDisplay}`. Verify with `node --check` on the extracted script tag.

2. Event listener leak — render() re-attaches click handler to every row on every call

Priority: 7/10 Confidence: 10/10 JavaScript Performance FIXED
J
Jason
JavaScript Specialist

Why this is a bug:

Inside render(), the code called document.querySelectorAll('tr.summary-row').forEach(tr => tr.addEventListener('click', ...)). Because render() is invoked on every filter change, sort, and row expand, the handlers accumulate across re-renders. With 583 listings × many user interactions in a session, the GC eventually keeps thousands of detached-but-referenced DOM listeners alive. More immediate symptom: clicking an expand-row that was previously expanded fires multiple toggle handlers on the freshly-rendered row, causing flicker.

Suggested fix:

Use event delegation: attach a single listener to #listings-body once at startup, look up the closest tr.summary-row ancestor from event.target, and toggle expand state on its data-id.

Fix Prompt:

In dashboard_template.html: remove the `document.querySelectorAll('tr.summary-row').forEach(tr => tr.addEventListener('click', ...))` block from inside render(). Outside render() and after the IIFE has declared `expanded` and `render`, add: `document.getElementById('listings-body').addEventListener('click', (ev) => { const tr = ev.target.closest('tr.summary-row'); if (!tr) return; const id = tr.dataset.id; if (expanded.has(id)) expanded.delete(id); else expanded.add(id); render(); });`

3. HTML injection in user-controlled string interpolations

Priority: 5/10 Confidence: 9/10 Security XSS FIXED
T
Tariq
Security & OWASP

Why this is a bug:

The row template interpolated ${l.title}, ${l.street}, ${l.neighborhood}, ${l.source}, ${l.condition}, ${l.id}, and ${l.image} directly into the HTML string passed to tbody.innerHTML. Because the listings come from scraped sources (Le Figaro, Bien'ici, Green Acres) where field values include arbitrary free-form text, a malicious or even accidentally-malformed listing title with <script> or quote-breaking content could execute or break the page. Likelihood is low (most scrapes are well-formed), but the cost of preventing it is trivial.

Suggested fix:

Add a small esc() helper that HTML-escapes & < > " ', and wrap every user-controlled interpolation with it. The renderAnalysis() function already does this for the analysis blob — reuse the same approach for cell content.

Fix Prompt:

In dashboard_template.html, add a helper above renderAnalysis():\n  function esc(s) {\n    if (s == null) return '';\n    return String(s).replace(/&/g,'&amp;').replace(/</g,'&lt;').replace(/>/g,'&gt;').replace(/"/g,'&quot;').replace(/'/g,'&#39;');\n  }\nThen wrap every l.title, l.street, l.neighborhood, l.source, l.condition, l.id, l.exposure, l.image, l.arr_name reference inside template literals with esc(). Leave numeric fields (price_eur, surface_m2, etc.) unwrapped since they're not user-controlled strings.

4. Listing thumbnails missing loading="lazy" — slows initial render

Priority: 4/10 Confidence: 9/10 Performance FIXED
M
Marcus
Performance & Networking

Why this is a bug:

The expanded-row meta panel renders <img src="${l.image}" ...> without loading="lazy". With 583 listings (and growing), each expand triggers a fresh image fetch from remote sources (apimo.pro, Green Acres, Bien'ici). When a user expands many rows or the browser pre-fetches them, this creates dozens of concurrent network requests against third-party CDNs that may rate-limit. Native lazy loading defers fetch until the image is near the viewport.

Suggested fix:

Add loading="lazy" to the <img> tag. Also add alt="${esc(l.title)}" for screen-reader accessibility (the original had alt="").

Fix Prompt:

In dashboard_template.html, in the expanded_html template literal, change:\n  <img class="image" src="${l.image}" alt="" onerror="this.style.display='none'" />\nto:\n  <img class="image" src="${esc(l.image)}" alt="${esc(l.title || '')}" loading="lazy" onerror="this.style.display='none'" />
